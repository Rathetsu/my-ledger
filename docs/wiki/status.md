# Status

**Current state: P7 complete.** Flexible debts and the deterministic payoff planner shipped. Flexible debts are a per-user table (`flexible_debts`, integer minor-unit money columns + the shared `currency` enum) with full CRUD; the balance is DERIVED (`original_minor` + signed adjustments − `debt_payment` sums), never stored. Payments record/reverse through a plain `userId`-taking module (`lib/debts/payments.ts`) with thin `'use server'` wrappers, mirroring the P3 confirm rail: `postDebtPayment` posts a negative `debt_payment` in the debt's currency into a same-currency account, and both post and reverse enforce the archived-account write-freeze (`account.archivedAt` inside the tx on post; `isAccountArchived` on reverse) — unit-tested on both paths. The planner (`lib/planner/`, pure `buildPlan(input): PlanResult`) owns 100% of displayed numbers: `surplus = guaranteed income − bills − installment obligations − variable-spend estimate` in home currency at live rates, then debt-first allocation — deadline just-in-time payments (`jitPayment`, apr/12) and defined minimums as obligations (scheduled even past surplus), then ASAP avalanche by APR descending; deadlined debts release slack into `unallocatedMinor` (P8's wishlist draws from it); currency-aware funding gaps carry live-rate transfer suggestions; installments with `apr ≥ 15` are flagged, never avalanched. The variable-spend estimate consumes P6's `variableSpendActuals` feed: essentials baseline until ≥3 complete months of non-`one_off` actuals exist, then the trailing-3-month mean per currency. `/debts` is reached from the More hub (the 7-slot bottom bar is full); `/plan` renders the payment timeline, per-debt payoff badges, funding-gap callouts, the spend-estimate source label, high-APR flags, and a disabled "AI advisor is off." slot that P9 swaps for cached advice. Gated green by Vitest (173, DB-backed against the Neon dev branch) and Playwright (the new debts+plan flow plus the archived-writes invariant spec).

**Plan reconciliation note (P7).** The written plan ([07-debts-and-planner.md](../superpowers/plans/07-debts-and-planner.md)) predated the P4–P6 conventions and was reconciled to current standards during execution: money columns are `integer` (not `bigint`) using the shared `currencyEnum` (C3); the zod schemas moved to `lib/actions/schemas.ts` so the `'use server'` `lib/actions/debts.ts` exports only async functions — the exact non-async-export 404 class the P6 e2e caught (C8); the payment DB logic was extracted to a plain `userId`-taking module so the write-freeze is unit-testable, mirroring `lib/occurrences/confirm.ts` + `lib/actions/occurrences.ts`; the archived-account guard was added to BOTH record and reverse (the plan omitted it entirely) (C1); `/debts` was wired into the More hub rather than an 8th tab (C4); and the e2e was made self-contained + `Date.now()`-stamped against the unscoped, accumulating dev lists (C6). Deliberately NOT changed: `flexible_debts` has no `account_id` (payments choose an account at pay-time), so debts are correctly absent from `archiveBlockers` — archiving an account cannot strand a debt. A whole-branch adversarial review returned 0 critical / 0 important; the e2e also exposed a real O(n) latency bug (`buildPlanInput` serialized one HTTP round trip per account/debt against the dev DB's ~434 accumulated accounts, ~108s), fixed by parallelizing the independent lookups with `Promise.all` (result-preserving). Remaining minors (created_at `timestamptz`, an aggregate-SUM balance query + `(source_type, source_id)` index, hand-typed currency options) are tracked as follow-ups, none merge-blocking.

**P6 (expenses & insights).** Expense categories are a per-user lookup table (`expense_categories`) with full CRUD + a mobile categories screen (create / edit via a native `<details>` disclosure / delete; deleting a category nulls its expenses' `category_id` in the same DB transaction, never deleting the expense). The P2 expense form gained a native category picker (`categoryId` submitted as FormData) alongside the pre-existing `one_off` checkbox; `postTransaction` resolves the client-supplied `categoryId` through a user-ownership check (foreign/absent → null) and never attaches a category to income. `/expenses` lists a month's expenses with month + category filters (both URL params range/format-guarded so a crafted value degrades to "no filter"/current month instead of a Postgres cast 500). `/expenses/insights` renders per-currency Recharts sections (spend-by-category stacked bar + monthly trend), never mixing currencies (spec §5.6), each with a plain-text totals list for accessibility. P7's planner feed `variableSpendActuals` (excludes `one_off` and the current partial month) and the pure `addPeriods` / `pivotByCategory` / `trendSeries` helpers are unit-tested; the two grouped-spend queries are DB-backed tested with window-boundary assertions. Gated green by Vitest (133, DB-backed against the Neon dev branch) and Playwright (the expenses + insights flow).

**Post-P6 review.** A 5-dimension adversarial whole-branch review (each finding independently verified) returned 0 critical / 0 important; the substantive minors were fixed: an out-of-range `?month=` guard (was a self-inflicted 500), the previously-dead `updateCategory` wired to a real edit affordance, and the category write-paths extracted into a testable non-`'use server'` module (`lib/expense-categories.ts`) with DB-backed tests for the `categoryId` trust boundary and cross-user delete isolation. The Task 9 e2e also caught a real runtime bug that `tsc` + a unit test had both missed: `lib/actions/expense-categories.ts` exported a zod schema from a `'use server'` file (Next forbids non-async exports), 404-ing the categories screen — fixed by moving the schema to `lib/actions/schemas.ts`.

**P5 (installments).** Count-based installments are the third and last occurrence kind, riding the same P3/P4 rails: the `installments` table + CRUD (create sets `remaining_count = total_count`; edits rewrite pending occurrences atomically and force `active=false` at 0 remaining), `housekeeping()` now also generates `kind='installment'` occurrences but only while `active` and `remaining_count > 0`, never before `start_date`, and never more future occurrences than payments left. Confirming posts an `installment_payment` (outflow) AND decrements `remaining_count` in the same DB transaction — the payment and the countdown move together or not at all; hitting 0 flips `active=false` and deletes leftover pending occurrences, and un-confirm increments the count back and reactivates. The dashboard attention list shows installments due within 7 days or overdue; installments get their own screens (progress derived as `total − remaining`, next due date) + an "Inst" bottom tab (now 7 tabs). The archived-account write-freeze holds across installment create/edit/confirm (`loadSource` joins `accounts` and rejects `archivedAt`), and `archiveBlockers()` now lists active installments too. Gated green by Vitest (91, DB-backed against the Neon dev branch) and Playwright (full 15-spec suite; the load-sensitive ledger/income flakes pass on re-run/in isolation).

**Plan drift note (P5):** the written plan ([05-installments.md](../superpowers/plans/05-installments.md)) predated the P3/P4 executor-param and archived-guard refactors. Three verbatim blocks would have regressed known invariants and were reconciled against current code: `loadDefinition`/`updateInstallment` keep the 3-arg atomic `rewritePendingOccurrences(..., tx)`; `loadSource` mirrors the bill case's account-join + archived guard; and `archiveBlockers` (omitted entirely from the plan's tasks) gained its active-installments clause.

**Post-P5 review remediation (branch `fix/review-remediation`).** A full-project multi-agent review found the archived-account write-freeze was enforced on every *insert* path but bypassed on every other write verb. Closed by moving the guard to the shared seams via a single `isAccountArchived(userId, accountId)` primitive in `lib/db/queries.ts`: plain-transaction edit/delete (`loadOwnedPlainRow`), `unconfirmOccurrence`, `deleteTransferGroup`, and income/bill reactivation now all reject writes into an archived account. Also: `parseToMinor` caps at the int4 max and validates comma grouping; `getRates` validates the FX payload before persisting (no NaN-poisoning); the attention list hides archived-account occurrences; installment completion clears overdue (not just pending) leftovers; a sign-out control + redirect-if-authed were added; false-green tests now drive the real code (income retraction extracted to a helper, `queries.test.ts` DB-backed); and the three definition kinds share `lib/actions/definitions.ts` (`ownedActiveAccount`/`parseAmount`/`ActionResult`/`NotFoundError`) so the write-freeze guard lives in one place. Gated by Vitest (109, DB-backed) and Playwright (all specs pass in isolation; full-suite failures are the documented load flakes).
